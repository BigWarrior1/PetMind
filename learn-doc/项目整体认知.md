# PetMind 项目整体认知

## 项目最终形态

**上线后的系统**：
```
用户打开网页
    ↓
登录/注册账号
    ↓
添加自己的宠物信息（狗狗/猫咪，品种，年龄，体重...）
    ↓
进入问答界面，输入问题或上传图片
    ↓
AI回答："狗狗咳嗽可能的原因有..." + 引用来源
    ↓
如果情况危急，弹出警示："建议立即就医"
```

**技术效果**：
- 用户在网页提问 → AI基于兽医知识库回答（不是胡编）
- 可以传宠物皮肤照片让AI看图分析
- 回答会标注来源（比如引用了《犬瘟热防治指南》第3页）

---

## 核心功能

| 功能 | 说明 |
|------|------|
| **RAG智能问答** | 基于本地知识库回答，不是胡编 |
| **多模态分析** | 上传图片，AI分析宠物健康问题 |
| **就医警示** | 危急情况提醒用户去看兽医 |
| **宠物档案** | 管理多只宠物的健康信息 |
| **对话历史** | 记录每次问答，可回看 |

---

## 项目难点

| 难点 | 说明 | 应对 |
|------|------|------|
| **RAG效果调优** | 检索不到正确的知识 → 回答质量差 | 设计好的分词策略、选择合适的Top-K |
| **多轮对话** | 问"狗狗吐了" → AI追问"吐了几次？什么颜色？" | 设计主动追问的Prompt |
| **幻觉问题** | 即使RAG，AI仍可能胡编 | 接入安全风控、限制回答范围 |
| **模型响应速度** | LLM调用有延迟 | 优化检索、添加loading状态 |

## 开发建议

### 最有效的开发方式

| 建议 | 说明 |
|------|------|
| **任务式请求** | "帮我搭建 FastAPI 项目结构" / "帮我写 RAG 检索代码" |
| **分模块开发** | 不要一次做整个系统，一个模块一个模块来 |
| **遇到问题及时问** | 报错信息发给我，我帮你分析 |
| **定期备份代码** | 重要节点让我帮你 commit |

### 不好的方式

| 方式 | 问题 |
|------|------|
| ❌ "帮我做完这个项目" | 太大，会迷失方向 |
| ❌ 一次性扔几百行代码 | 会遗漏问题 |
| ❌ 不保存代码就继续 | 可能丢失工作 |

### 建议的开发节奏

1. 你告诉我今天做什么模块
2. 我帮你写代码、解释代码
3. 有问题随时问
4. 做完一块，测试一块，再做下一块
